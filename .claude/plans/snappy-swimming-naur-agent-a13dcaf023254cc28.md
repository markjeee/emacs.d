### Loading sequence (init.el)

1. Version check (27.1+), GC threshold bump (128 MB), process tuning
2. Bootstrap: `init-utils` -> `init-site-lisp` -> `init-elpa` (package.el) -> `init-exec-path`
3. Optional user preload: `lisp/init-preload-local.el`
4. Feature modules: 79 `lisp/init-*.el` files loaded via `require`
5. Optional user config: `lisp/init-local.el` (gitignored, personal settings)
6. `custom.el` loaded last (auto-generated by `M-x customize`)

### Key subsystems

- **Completion:** Vertico + Consult + Embark + Marginalia (minibuffer); Corfu + Orderless (in-buffer)
- **Syntax checking:** Flymake with various backends (not Flycheck)
- **LSP:** Eglot
- **VCS:** Magit (full-frame), Forge, diff-hl
- **Projects:** Projectile
- **Tree-sitter:** Conditional on Emacs 29+ with treesit available

### Package management

Uses **package.el** (not use-package or straight.el). Two key functions in `lisp/init-elpa.el`:

- `require-package` -- install a package or error
- `maybe-require-package` -- install a package or silently return nil

Packages install into version-specific `elpa-X.Y/` directories to avoid bytecode incompatibility. MELPA is the sole third-party archive.

## 5. Environment

**Prerequisites:**
- Emacs 27.1+ (latest stable recommended; config degrades gracefully on older versions)
- Git (for Magit, version control integration)
- Internet access on first startup (for MELPA package downloads)

**Environment variables:**
- `EMACS` -- path to Emacs binary (used by `test-startup.sh`; defaults to `emacs`)

**Platform detection:**
- `*is-a-mac*` constant (defined in `init.el`) guards macOS-specific code throughout the config
- `*spell-check-support-enabled*` constant controls spell-check loading (default: nil)

## 6. Code Style

- **Indentation:** Spaces only, no tabs (enforced by `.dir-locals.el`)
- **Lexical binding:** Every `init-*.el` file has `lexical-binding: t` in the file header
- **Module pattern:** Each `lisp/init-*.el` ends with `(provide 'init-FEATURE)`
- **Deferred setup:** Use `after-init-hook` or `with-eval-after-load` for deferred initialization
- **Minor mode lighters:** Use `diminish` to hide minor mode lighters
- **Namespace:** Custom functions/variables use the `sanityinc/` prefix
- **Prefer built-in:** Favor built-in Emacs facilities over third-party when reasonable
- **Soft dependencies:** Use `maybe-require-package` for optional/nice-to-have packages so startup does not break if they are unavailable
- **Version guards:** Conditional feature loading based on Emacs version using `version<`, `fboundp`, etc.
- **No byte-compile:** `init.el` has `no-byte-compile: t` in its local variables

## 7. Testing

**Test command:**
```sh
./test-startup.sh
```

**What it tests:** Batch-mode startup smoke test -- loads `init.el`, runs `after-init-hook`, and verifies Emacs starts without errors.

**CI matrix (`.github/workflows/test.yml`):**
- Emacs versions: 27.1, 27.2, 28.1, 28.2, 29.1, 29.3, 29.4, 30.1
- Emacs snapshot (experimental, allowed to fail)
- Runs on: `ubuntu-latest`
- CI ignores changes to `**.md` files

**There is no unit test framework, linter, or build step.** The smoke test is the sole automated validation.

### aide-operator reference

### Levels of Coding Control

| Level | Description | When to use |
|-------|-------------|-------------|
| **L1 -- Spec only** | Describe what to build; agent plans and implements | New feature modules, large refactors |
| **L2 -- Guided** | Provide direction and constraints; agent fills in details | Adding a new `init-*.el` module, updating package configs |
| **L3 -- Pair** | Work together interactively, reviewing each step | Debugging startup issues, tricky version-conditional code |
| **L4 -- Surgical** | Point at exact code; agent makes precise edit | Fixing a specific keybinding, adding a `require-package` call |
| **L5 -- Review only** | Agent reviews code, suggests changes, does not edit | Auditing module conventions, checking for style compliance |

## 11. Gotchas

- **No build/lint step:** The only validation is `test-startup.sh`. If you break startup, the smoke test catches it; nothing else will.
- **Version-specific elpa dirs:** Packages install to `elpa-X.Y/` (e.g., `elpa-29.4/`). These are gitignored. Do not reference absolute elpa paths.
- **init-local.el is gitignored:** It exists on disk but is not in the repo. Do not modify it expecting changes to be tracked.
- **custom.el is gitignored:** Same as above. It is loaded last and can override anything.
- **Module load order matters:** The require chain in `init.el` is sequential and order-dependent. Inserting a new `require` in the wrong position can cause missing-dependency errors.
- **CI skips .md changes:** The workflow has `paths-ignore: '**.md'`. Changes to markdown files alone will not trigger CI.
- **maybe-require-package returns nil on failure:** Code after a `maybe-require-package` call must handle the nil case (typically via `when`).
- **Platform conditionals:** macOS-specific code is gated by `*is-a-mac*`. Do not assume macOS APIs are available unconditionally.
- **Upstream is `trunk`, not `origin`:** The `trunk` remote points to `purcell/emacs.d`; `origin` points to the fork `markjeee/emacs.d`.
- **No use-package or straight.el:** This config uses raw `package.el`. Do not introduce `use-package` macros.

## Design Decisions

1. **All CLAUDE.md content migrates to AGENTS.md:** The existing CLAUDE.md content maps cleanly to sections 1-7 and 11 of the AGENTS.md template. No content needs to remain in CLAUDE.md.

2. **Module count updated to 79:** Actual count from `ls lisp/init-*.el | wc -l` is 79, not "~80" as stated in the original CLAUDE.md. The AGENTS.md uses the precise count.

3. **Gotchas section is enriched:** Beyond what was in CLAUDE.md, the Gotchas section includes practical pitfalls discovered during exploration (upstream remote naming, CI .md skip, no use-package, etc.).

4. **Key Files uses a table format:** More scannable than prose for an agent that needs to quickly find the right file.

5. **Commands section includes Emacs-internal commands:** Package refresh and update commands are included even though they run inside Emacs, because an agent may need to instruct the user about them.

6. **RULES.md is intentionally empty:** No project-specific rules were identified. The file exists as a placeholder for future rules.

7. **SKILLS.md is minimal:** Only registers the two AIDE skills. Can be extended as more skills are identified.

8. **Fork/upstream context included:** The Agent Architecture and Project sections note both remotes, which is important for an agent that might need to push or pull.
